Seller Product Management — What's in scope?

Let's define the MVP boundaries before we design anything. Which of these best describes what you want to build first?

A

Core CRUD Only

Create, edit, delete, and list products. Basic fields: title, description, price, images, stock. No variants, no categories. Ship fast, iterate later.

Pros

  • Fastest to build
  • Unblocks buyer storefront immediately
  • Easy to test end-to-end

Cons

  • No variant support (size, color)
  • No category organisation
B

CRUD + Variants + Categories

Everything in A, plus Medusa's native variant system (options like Size/Color), category tagging, and a richer product form.

Pros

  • Production-ready product data model
  • Matches how Medusa natively works
  • Buyer storefront can filter by category

Cons

  • More complex form UI
  • Takes 2-3x longer to build
C

CRUD + Status Workflow

Everything in A, plus product status (Draft → Pending Admin Approval → Published). Admin must approve before a product goes live on the storefront.

Pros

  • Platform control over what's listed
  • Connects to admin workflow
  • Real marketplace feel

Cons

  • Requires admin product queue UI
  • More backend state machine
